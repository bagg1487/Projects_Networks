## Анализ Latency и Throughput

### Характер роста Latency
1. **REST сервис**:
   - Низкая нагрузка (до 200 RPS): латентность стабильна (~40ms)
   - Средняя нагрузка (200-450 RPS): **линейный рост** латентности
   - Высокая нагрузка (>450 RPS): **экспоненциальный рост** латентности

2. **gRPC сервис**:
   - Низкая нагрузка (до 3000 RPS): латентность почти не растет (~5ms)
   - Средняя нагрузка (3000-5000 RPS): слабый линейный рост
   - Высокая нагрузка (>5200 RPS): экспоненциальный рост

### Точки насыщения (Saturation Point)

**REST**:
- Точка насыщения: **450-500 RPS**
- Признаки насыщения:
  - RPS перестает расти (487 → 512, +5%)
  - P99 латентность резко возрастает (445ms → 3521ms, +690%)
  - CPU утилизация достигает 94%

**gRPC**:
- Точка насыщения: **5000-5200 RPS**
- Признаки насыщения:
  - RPS растет незначительно (5234 → 5412, +3.4%)
  - P99 латентность резко возрастает (45.8ms → 512ms, +1018%)
  - CPU утилизация достигает 96%

## Выводы

1. **Точка насыщения сервиса comments-s14**:
   - REST: ~470 RPS (при 0.5 CPU)
   - gRPC: ~5100 RPS (при 0.5 CPU)

2. **Характер роста Latency**:
   - До 60% от насыщения: латентность растет линейно
   - После 80% от насыщения: латентность растет экспоненциально
   - Это соответствует теоретическим моделям очередей (M/M/1)

3. **Почему gRPC значительно быстрее**:
   - Protobuf сериализация в 5-10 раз быстрее JSON на ограниченном CPU
   - HTTP/2 мультиплексирование эффективнее использует соединения
   - Бинарный протокол уменьшает overhead парсинга

4. **Практические рекомендации для production**:
   - Держать нагрузку не выше 60-70% от saturation point
   - Для REST: target RPS ≤ 300
   - Для gRPC: target RPS ≤ 3500
   - Настроить rate limiting на этих уровнях
   - Использовать горизонтальное масштабирование при превышении

## Ответы на вопросы

**Q: Какой перцентиль лучше отражает пользовательский опыт?**  
A: P99 и P95, так как они показывают опыт 1% и 5% "медленных" пользователей. Среднее значение (avg) может быть обманчивым при наличии выбросов.

**Q: Что делать, если P99 latency высокая, но средняя низкая?**  
A: Это указывает на проблемы с "хвостами" (tail latency):
- GC паузы (для managed языков)
- Blocking операции (I/O, locks)
- "Неравномерное" распределение запросов
- Рекомендуется использовать инструменты tracing (Jaeger, Zipkin)

**Q: Как масштабировать comments-s14 при достижении насыщения?**  
A: 
1. Добавить горизонтальные реплики в Kubernetes (`kubectl scale deployment comments-app --replicas=3`)
2. Настроить HPA (Horizontal Pod Autoscaler) с метриками CPU/RPS
3. Использовать load balancer перед сервисом

**Q: Почему при 1000 соединениях RPS почти не растет?**  
A: Из-за ограниченного CPU (0.5 ядра):
- Увеличивается contention за ресурсы
- Растут очереди запросов
- Больше времени тратится на context switching
- Это классический признак saturation

## Закон Литтла (Little's Law) для comments-s14

При насыщении REST (470 RPS, latency 0.445s):  
`Число запросов в системе = 470 * 0.445 = 209`  
То есть 209 запросов одновременно обрабатываются или ждут в очереди.

При насыщении gRPC (5100 RPS, latency 0.0458s):  
`Число запросов в системе = 5100 * 0.0458 = 233`  
Несмотря на гораздо больший RPS, количество параллельных запросов примерно такое же из-за меньшей латентности.
